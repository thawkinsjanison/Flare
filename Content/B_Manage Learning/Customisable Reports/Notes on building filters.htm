Building filters

You can set up rules to precisely define parts of a CLS entity that you want to report on. These rules are called filters. They can very precisely select members of an entity because they are combined using Boolean logic, which can also be nested. This will be best understood by someone who is comfortable with:

  • Boolean operations (AND, OR, NOT),
  • its language (e.g. “!” means “NOT”), and
  • how nesting limits scope.

Clear and detailed instructions about how to set these up are found at Building Filters, including a very simple introduction for beginers. You will find this filter finctionality in a number of places in the CLS.

Using filters you can also make a Report pause while it is running in order to allow a filter value to be entered on the spot. You use the filter's Control checkbox, which is explained at point 13 in Filters.

Note that an alternative to using filters is to export the results of your Report to an Excel file. In Excel you can do filtering and manipulation of the data in an application that many people are familiar with.

Note also that in addition to the filters that you specify here, the system will append further filters based on Security roles. This makes the customisable reporting functionality available to a wider range of Users (as it doesn’t need to be restricted to administrator Users). It also leverages existing security settings about who can see who, for example, no special or separate configuration are required for a Supervisor to only see their staff, or a Teacher to only see their students. This makes it impossible for someone to accidentally be granted too many permissions at the reporting level and improves performance because most Users will only be able to see a subset of the total results. The security filtering applied is the same as for the User List Form (under Manage Users menu item) – the same Security filtering also applies to Reports on enrolment entities.
